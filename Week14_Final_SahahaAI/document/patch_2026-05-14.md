## 2. 보안 패치

### 2.1 SECRET_KEY 환경변수 강제화

- **파일:** [config.py](../config.py)
- **무엇을:** 기존 하드코딩 기본값(`"saha-chatbot-secret-key-change-in-production"`)을 제거하고, `.env`에 미설정 시 `ValueError`로 서버 시작을 차단.
- **왜:** Flask 세션 쿠키는 `SECRET_KEY`로 서명된다. 하드코딩 기본값이 노출되면 누구나 세션을 위조하여 다른 사용자로 가장할 수 있다.
- **어떻게:**
  ```python
  SECRET_KEY = os.getenv("SECRET_KEY")
  if not SECRET_KEY:
      raise ValueError("SECRET_KEY를 .env에 설정해주세요 (세션 위조 방지)")
  ```

### 2.2 관리자 API 인증 (`/api/stats`)

- **파일:** [app.py](../app.py)
- **무엇을:** `/api/stats` 엔드포인트에 `@require_admin` 데코레이터를 추가하여 `X-Admin-Key` 헤더 검증을 강제.
- **왜:** 기존에는 누구나 호출하여 `raw_pages`/`chunks`/`embedded`/`conversations` 카운트와 같은 시스템 내부 정보를 조회할 수 있었다. 정찰 단계에서 시스템 규모를 노출하는 정보 누출이다.
- **어떻게:** `ADMIN_API_KEY` 환경변수로 키를 정의하고, 요청 헤더 `X-Admin-Key`와 정확히 일치하지 않으면 401 반환. 키 미설정 시 503으로 엔드포인트 자체 비활성화.

### 2.3 에러 메시지 일반화

- **파일:** [app.py](../app.py)
- **무엇을:** `/api/stats` 예외 응답에서 `str(e)` 원본 노출을 제거하고 "시스템 오류가 발생했습니다"로 일반화. 상세 스택 트레이스는 서버 로그에만 기록.
- **왜:** Supabase URL이나 라이브러리 내부 경로가 에러 메시지에 포함될 수 있어 인프라 정보 유출 위험이 있었다.

### 2.4 CORS / X-Frame-Options / CSP

- **파일:** [app.py](../app.py)
- **무엇을:**
  - `flask-cors`로 `/api/*` 경로의 허용 출처를 `CORS_ALLOWED_ORIGINS` env로 제한.
  - 모든 응답에 `X-Frame-Options: SAMEORIGIN`, `Content-Security-Policy: frame-ancestors 'self' <허용출처>`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin` 헤더 부착.
- **왜:** `/widget` 엔드포인트는 사하구청 홈페이지 iframe 삽입을 의도한 것인데, 출처 제한이 없으면 악성 사이트가 위젯을 임베딩해 클릭재킹 공격을 수행할 수 있다.

### 2.5 Rate Limiting

- **파일:** [app.py](../app.py)
- **무엇을:** `flask-limiter`를 적용하여 `/api/chat`은 IP당 분당 10회(`RATE_LIMIT_CHAT` env), `/api/stats`는 분당 30회로 제한.
- **왜:** Groq 무료 티어의 분당 30회 제한을 초과하면 전체 사용자가 챗봇을 사용하지 못한다. 또한 무차별 호출로 인한 비용 폭증·DoS도 방지.

### 2.6 Supabase RLS 정책

- **파일:** [setup_supabase.sql](../setup_supabase.sql)
- **무엇을:** 모든 테이블에 RLS를 활성화하고 역할별로 정책을 분리.
  - `documents`, `processed_chunks`: anon은 SELECT만 가능 (챗봇 검색용)
  - `raw_pages`: anon 접근 차단 (관리 작업 전용)
  - `conversation_logs`: anon은 INSERT/SELECT/DELETE만, UPDATE 금지 (append-only)
  - `service_role` 키는 RLS를 자동 우회하므로 관리 작업은 별도 정책 없이 동작
- **왜:** Supabase Security Advisor에서 CRITICAL 6건이 보고된 핵심 원인이었다. RLS가 꺼져 있으면 anon 키 하나로 전체 테이블 읽기/삭제가 가능하다. 특히 `conversation_logs`에는 사용자 질문이 평문으로 저장되므로 가장 위험.
- **부수 작업:** [database_db/__init__.py](../database_db/__init__.py)에 `get_supabase(admin=...)` 팩토리를 만들어, 챗봇 요청은 anon 키로, 크롤링·TTL 같은 관리 작업은 `SUPABASE_SERVICE_KEY` 기반의 service_role 키로 분리 사용하도록 클라이언트를 이중화.

### 2.7 match_documents 함수 보안 강화

- **파일:** [setup_supabase.sql](../setup_supabase.sql)
- **무엇을:**
  - `SECURITY DEFINER` 함수에 `set search_path = public, pg_catalog` 추가
  - `public` 역할의 실행 권한 회수 후 `anon`/`authenticated`에만 명시적 부여
- **왜:** Supabase Security Advisor의 "Function Search Path Mutable" / "Public Can Execute SECURITY DEFINER Function" 경고를 해소. SECURITY DEFINER 함수에서 `search_path`가 가변이면 동일 이름의 악의적 함수를 다른 스키마에 심어 권한 상승 공격이 가능하다.

## 3. 성능 개선

### 3.1 벡터 검색을 match_documents() RPC로 전환

- **파일:** [database_db/vector_store.py](../database_db/vector_store.py)
- **무엇을:** 기존에 `documents` 테이블 전체를 클라이언트로 가져와 Python에서 코사인 유사도를 계산하던 방식을 폐기하고, Supabase에 정의된 `match_documents()` RPC를 호출하여 서버사이드 pgvector + HNSW 인덱스로 검색.
- **왜:**
  1. **보안:** 매 검색마다 전체 임베딩 벡터(수 MB 이상)가 네트워크로 전송되었음. 임베딩은 학습 데이터를 역추론할 수 있는 민감 정보다.
  2. **성능:** 데이터가 늘어날수록 검색 시간이 선형 증가. HNSW 인덱스 기반 RPC는 로그 시간에 가깝다.
- **결과:** 클라이언트는 384차원 query embedding 1개만 전송하고, 서버는 정렬된 상위 결과만 반환.

### 3.2 Supabase 클라이언트 공유

- **파일:** [database_db/__init__.py](../database_db/__init__.py) (신규)
- **무엇을:** `Database`와 `VectorStore`가 각각 `create_client()`로 독립적인 Supabase 연결을 만들던 것을 모듈 레벨 싱글턴으로 통합.
- **왜:** 매 인스턴스마다 TCP 연결·세션 핸드셰이크 비용이 발생했고, 메모리 사용도 두 배였다.
- **확장:** 동일 모듈에서 `admin=True` 인자로 service_role 클라이언트를 별도 캐싱하여, 권한별 클라이언트 분리도 한 군데에서 관리.

### 3.3 챗봇/DB/VectorStore 싱글턴 재사용

- **파일:** [app.py](../app.py)
- **무엇을:** 기존 `/api/stats`는 호출마다 `Database()`와 `VectorStore()`를 신규 생성하여, 매 호출마다 임베딩 모델(약 80MB)을 디스크에서 다시 로딩(10초+)했다. `get_chatbot()`/`get_db()`/`get_vector_store()` 싱글턴으로 재사용.
- **왜:** 단순 통계 조회 한 번에 10초 이상 응답이 지연되는 비정상 동작.

### 3.4 BFS 큐 deque 전환

- **파일:** [crawler/saha_crawler.py](../crawler/saha_crawler.py)
- **무엇을:** `queue.pop(0)`(O(N))을 `collections.deque.popleft()`(O(1))로 교체.
- **왜:** 큐 길이가 늘어날수록 pop 비용이 증가. 페이지 수가 많아지면 크롤링 후반부에서 체감 가능한 지연.

### 3.5 메타데이터 태깅 배치 처리

- **파일:** [processor/metadata_tagger.py](../processor/metadata_tagger.py)
- **무엇을:** 청크 1개당 1회 LLM 호출 + 25초 sleep으로 100청크에 약 42분 걸리던 구조를, 5개 청크를 단일 프롬프트에 묶어 1회 호출(`BATCH_TAGGING_PROMPT`) + 2초 sleep으로 변경. JSON 파싱 실패 시 단건 폴백.
- **왜:** Groq 무료 티어 분당 30회 한도 안에서 처리량을 극대화. 100청크 기준 약 9분으로 단축(약 78% 감소).

## 4. 기능 추가

### 4.1 LLM 응답 PII 출력단 마스킹

- **파일:** [chatbot/conversation.py](../chatbot/conversation.py)
- **무엇을:** 기존 PII 차단은 사용자 입력만 검사했다. 크롤링 데이터에 우연히 포함된 개인정보(예: 담당자 휴대폰)가 LLM 응답에 그대로 노출될 수 있어, `mask_personal_info()` 헬퍼를 추가하고 응답 단계에서도 정규식 패턴으로 마스킹(`[MASKED:전화번호]` 등).
- **왜:** 입출력 양방향 PII 보호. 카드번호 패턴(16자리)도 추가.
- **구조:** `detect_personal_info()`(탐지)와 `mask_personal_info()`(마스킹)를 모듈 함수로 분리하여 재사용 가능하게 했다.

### 4.2 역질문 감지를 [CLARIFICATION] 태그 기반으로

- **파일:** [chatbot/conversation.py](../chatbot/conversation.py)
- **무엇을:** 기존에는 답변에 `"어떤 분야"`, `"궁금하신가요?"` 같은 키워드가 들어 있는지로 역질문 여부를 판단했다. LLM 응답 표현이 조금만 달라져도 잘못 판정되는 문제가 있었다. 시스템 프롬프트에 "역질문 시 마지막 줄에 `[CLARIFICATION]`을 붙여라"는 지시를 추가하고, 응답에서 해당 태그를 우선 검사하도록 변경. 태그 누락 시 기존 키워드 휴리스틱으로 폴백.
- **왜:** LLM 의도를 명시적으로 신호화하여 표현 변형에 강건하게.

### 4.3 대화 이력 자동 TTL (30일)

- **파일:** [database_db/database.py](../database_db/database.py), [main.py](../main.py)
- **무엇을:** `cleanup_old_conversations(ttl_days)` 메서드를 추가하고, APScheduler에 매일 04:00 실행 작업으로 등록. TTL은 `CONVERSATION_TTL_DAYS` env(기본 30일)로 조정 가능.
- **왜:** `conversation_logs`가 영구 누적되면 개인정보보호법상 보관 기한 문제, 그리고 Supabase 무료 티어 용량 제약이 있다. append-only 정책상 자동 정리가 유일한 해법.

### 4.4 DB 기반 중복 청크 감지

- **파일:** [processor/data_cleaner.py](../processor/data_cleaner.py)
- **무엇을:** `DataCleaner(db=db)` 생성 시 DB에서 기존 `chunk_id` 전수를 한 번 조회하여 메모리 셋에 캐시. `process()`에서 동일 `chunk_id`는 임베딩 단계까지 진입하지 않고 스킵.
- **왜:** 기존 구현은 `_seen_hashes`가 인스턴스 메모리에만 존재해 재실행 시 초기화되었다. DB에서 최종적으로 upsert로 막히긴 했지만, 임베딩 연산(CPU 비용)을 헛수고로 소진하고 있었다.

### 4.5 크롤러 robots.txt 준수

- **파일:** [crawler/saha_crawler.py](../crawler/saha_crawler.py)
- **무엇을:** 크롤러 초기화 시 `urllib.robotparser`로 `/robots.txt`를 로딩하고, 각 URL fetch 전 `can_fetch()`로 허용 여부 검사. 차단된 URL은 로그를 남기고 스킵.
- **왜:** 공공기관 사이트라 실제로 차단당할 가능성은 낮지만, 정식 서비스로서 표준 크롤링 윤리를 준수하기 위함. 향후 다른 사이트로 확장할 때도 동일 코드로 안전.

## 8. 적용 절차

1. `pip install -r requirements.txt` (또는 `py -m pip install -r requirements.txt`)
2. `.env`에 신규 변수 추가 (`SECRET_KEY`, `ADMIN_API_KEY`, `SUPABASE_SERVICE_KEY`)
3. Supabase Dashboard → SQL Editor에서 갱신된 `setup_supabase.sql` 전체 재실행
4. `python main.py --mode web` 으로 서버 기동

| 항목 | 확인 방법 | 정상 응답 |
|------|----------|-----------|
| 보안 헤더 | `curl -I http://127.0.0.1:5000/` | `X-Frame-Options`, `CSP`, `X-Content-Type-Options` 헤더 존재 |
| Rate Limit | `/api/chat`을 1분에 11회 호출 | 11번째 호출에서 429 응답 |
| 관리자 인증 | `curl http://127.0.0.1:5000/api/stats` (헤더 없음) | 401 응답 |
| 관리자 정상 호출 | `curl -H "X-Admin-Key: <키>" http://127.0.0.1:5000/api/stats` | 통계 JSON 반환 |
| 입력 PII 차단 | 챗봇에 "내 번호는 010-1234-5678" | 경고 메시지, LLM 미호출 |
| 출력 PII 마스킹 | 응답 내 전화번호/이메일 | `[MASKED:전화번호]` 형태로 치환 |
| RLS 적용 | Supabase Advisors 페이지 | CRITICAL 0건 |
| TTL 스케줄러 | 서버 기동 로그 | "스케줄러 등록 (... / 대화 이력 30일 정리 04:00)" |
| 벡터 RPC | 챗봇에 일반 질문 | 응답 정상 + 로그에 "[검색결과] 유사도=..." 출력 |

## 10. 잔여 과제

- **thread-safety:** Flask 단일 프로세스 가정. 다중 워커(gunicorn/FastAPI) 전환 시 챗봇 싱글턴에 `threading.Lock` 또는 워커별 인스턴스 분리 필요.
- **silent failure UX:** RPC/태깅 실패 시 현재는 로그만 남기고 빈 결과를 반환. 사용자에게 "일부 검색 결과를 가져오지 못했습니다" 안내 추가 예정.
- **비정형 PII:** 정규식 패턴 외에 이름·주소 같은 비정형 정보는 미차단. 한국어 NER(KoELECTRA 등) 도입 검토.
- **하이브리드 검색 가중치 최적화:** BM25 + 벡터 점수 가중 합산 비율을 테스트셋 기반으로 정량 평가하여 확정.
- **비동기 서버:** Flask → FastAPI 전환으로 동시 요청 처리 성능 향상.
