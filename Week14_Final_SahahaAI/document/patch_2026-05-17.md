## 2. 1단계: NER 기반 비정형 PII 차단

### 2.1 문제 정의

5/14 패치에서 LLM 응답 PII 마스킹을 도입했지만, **정규식만으로는 한계**가 있었다.

- 정규식이 잘 잡는 것: 주민등록번호, 전화번호, 카드번호, 이메일 (패턴이 고정된 정형 PII)
- 정규식이 못 잡는 것: 이름(홍길동), 상세 주소(다대동 123-45번지) 같은 **비정형 PII**

비정형 PII는 의미를 이해해야 식별 가능하므로 머신러닝 기반 접근이 필요했다.

### 2.2 채택한 모델

`Leo97/KoELECTRA-small-v3-modu-ner`
- KoELECTRA(한국어 사전학습 모델) 기반 NER fine-tuning
- 크기: 약 50MB (small 버전 — 경량 + 빠른 추론)
- 라벨: 한국어 14개 엔티티(PS=인명, LC=지명, OG=기관, AF=물건/제품, DT=날짜 등)
- CPU 추론 시간: 약 100~300ms

### 2.3 핵심 설계

#### 2.3.1 정규식 + NER 병렬 마스킹

[chatbot/conversation.py](../chatbot/conversation.py)의 `mask_personal_info()`를 2단계 파이프라인으로 재구성.

```
사용자 입력 / LLM 응답
        ↓
[1단계] 정규식 (PERSONAL_INFO_PATTERNS)
   → 전화번호/주민번호/카드번호/이메일 → [MASKED:전화번호]
        ↓
[2단계] NER (NERPIIDetector)
   → 인명/지명/물건 → [MASKED:이름], [MASKED:주소], [MASKED:물건]
        ↓
   마스킹된 최종 텍스트
```

각 단계가 독립적이라 NER 모델 로딩 실패해도 정규식은 정상 동작 (graceful degradation).

#### 2.3.2 SAFE_TOKENS — 공공기관·지역명 보호

NER 모델은 "사하구청", "부산", "사하구" 같은 공식 명칭도 LOCATION/ORG로 감지한다. 그대로 두면 챗봇 답변이 "[MASKED:주소] 홈페이지에 따르면..."처럼 깨진다.

해결: [chatbot/pii_detector.py](../chatbot/pii_detector.py)에 화이트리스트 토큰 정의.

```python
SAFE_TOKENS = {
    "사하구", "사하구청", "부산", "부산광역시", "부산시",
    "대한민국", "한국",
}
```

이 단어는 NER이 감지해도 마스킹에서 제외.

#### 2.3.3 OG(기관) 라벨 자체 제외

`MASK_LABEL_MAP`에서 OG는 아예 마스킹 대상에서 뺐다. 행정 안내에서 기관명은 핵심 정보(예: "동구청 민원실에 문의하세요")이므로 마스킹하면 답변이 무용지물.

#### 2.3.4 score threshold 0.75

NER 모델의 모든 매칭이 정확한 것은 아니다. 신뢰도(score)가 낮은 매칭은 오탐(예: "전화" → 인명으로 잘못 분류)일 수 있어 임계값으로 필터링.

```
score < 0.75 → 무시
score >= 0.75 → 마스킹
```

테스트로 확인한 실제 점수:
- "홍길동" → score 0.81 (포착됨)
- "김철수" → score 0.98 (확실히 포착)
- "이순신" → score 0.98 (확실히 포착)

0.85로 시작했다가 "홍길동"이 누락되는 문제로 0.75로 낮춰 튜닝.

#### 2.3.5 싱글턴 + 사전 로딩

모델 로딩에 시간이 걸리므로 `NERPIIDetector`를 싱글턴(`__new__` 패턴)으로 만들고, `ChatBot.__init__`에서 사전 로딩하도록 했다. 첫 사용자 요청이 100~300ms씩 지연되는 일을 방지.

### 2.4 검증

| 입력 | 마스킹 결과 |
|---|---|
| `홍길동입니다` | `[MASKED:이름]입니다` |
| `김철수 씨가 사하구청에` | `[MASKED:이름] 씨가 사하구청에` (사하구청은 SAFE) |
| `부산 사하구 다대동 123-45번지... 010-1234-5678` | `부산 사하구 [MASKED:주소] 123-45번지... [MASKED:전화번호]` |
| `민원 신청은 어떻게 하나요?` | 변경 없음 (오탐 없음) |
| `이순신 장군의 동상` | `[MASKED:이름] 장군의 동상` |

### 2.5 새 의존성

- `transformers` (HuggingFace, NER 파이프라인)
- `torch` (PyTorch, 추론 백엔드)
- 모델 자동 다운로드: 약 50MB (`Leo97/KoELECTRA-small-v3-modu-ner`)

### 2.6 비용

- 메모리: 약 200MB 추가 (transformers + torch + 모델)
- 응답 시간: 약 100~300ms 증가 (CPU 추론)
- 첫 실행 시 모델 다운로드 시간 추가 (~30초)

## 3. 2단계: BM25 하이브리드 검색

### 3.1 문제 정의

기존 검색은 **벡터 유사도 검색**만 사용했다. 의미적으로 비슷한 문서를 잘 찾지만, 단점이 있다.

- 단점 1: **정확한 키워드 매칭에 약하다**
  - "전입신고"라는 정확한 단어가 들어간 문서를, 의미 유사도만으로는 1위로 못 찾는 경우 발생
- 단점 2: 짧은 쿼리에서 의미 모호함
  - "주민등록증" 같은 핵심 키워드는 의미 검색보다 단어 매칭이 더 적합

해결책: **벡터 검색 + BM25 키워드 검색 + 가중 합산**

### 3.2 BM25란

BM25(Best Matching 25)는 정보검색에서 표준으로 쓰이는 키워드 매칭 알고리즘.
- TF-IDF의 발전형
- 단어 빈도(TF) + 역문서빈도(IDF) + 문서 길이 보정
- 정확한 키워드 매칭에 강함

벡터 검색과의 차이:
- **벡터 검색**: "민원 신청"과 "행정 절차"가 의미 비슷하면 같이 검색됨 (의미 유사)
- **BM25**: "민원 신청"이라는 단어가 정확히 포함된 문서만 우선 (단어 매칭)

두 가지를 함께 쓰면 의미와 키워드 모두 잡는다.

### 3.3 핵심 설계

#### 3.3.1 BM25 인덱스 구조

[chatbot/bm25_index.py](../chatbot/bm25_index.py) 신규 작성. `BM25Index` 싱글턴 클래스.

- 서버 기동 시 Supabase `documents` 테이블 전체를 메모리로 로딩
- 각 문서를 Kiwi 형태소 분석기로 토큰화
- `rank_bm25.BM25Okapi`로 인덱스 구축
- 검색 시 쿼리도 같은 방식으로 토큰화 후 `get_scores()` 호출

#### 3.3.2 한국어 형태소 분석 — kiwipiepy

영어와 달리 한국어는 띄어쓰기만으로 단어를 못 나눈다 ("민원신청" 같은 합성어가 흔함). 형태소 분석기가 필수.

`kiwipiepy` 선택 이유:
- `konlpy`는 Java 의존성이 까다로움
- `kiwipiepy`는 C++ 빌드를 내장하지만 외부 의존성 없음
- 성능과 정확도 양호

#### 3.3.3 의미 토큰만 추출

조사("가", "는"), 어미("했다"), 기호 등은 검색에 의미 없는 노이즈.

```python
EXCLUDED_POS_PREFIXES = ("J", "E", "X", "S")  # 조사/어미/접사/기호

return [
    tok.form for tok in tokens
    if not tok.tag.startswith(EXCLUDED_POS_PREFIXES) and len(tok.form) > 1
]
```

명사·동사·형용사만 남기고, 1글자 토큰("의", "도")도 제외.

#### 3.3.4 벡터 + BM25 가중 합산

[chatbot/retriever.py](../chatbot/retriever.py)의 `_hybrid_combine()`에서:

```
점수 계산
    벡터 점수 (cosine similarity, 0~1)
    BM25 점수 (0~수십, 문서 길이/빈도에 따라 다름)
        ↓
정규화 (min-max → 0~1로 통일)
        ↓
가중 합산
    hybrid_score = 0.7 × vec_norm + 0.3 × bm25_norm
        ↓
재랭킹 → 상위 k개 반환
```

가중치는 환경변수로 조정 가능:
- `HYBRID_VECTOR_WEIGHT` (기본 0.7)
- `HYBRID_BM25_WEIGHT` (기본 0.3)
- `HYBRID_BM25_TOP_N` (BM25 후보 풀 크기, 기본 30)

#### 3.3.5 후보 풀 union

벡터 검색 상위 k×2개와 BM25 상위 30개의 **합집합**을 후보로 둔다. 두 알고리즘이 다른 문서를 1위로 꼽을 때 모두 후보에 포함시켜야 가중 합산 효과가 살아난다.

### 3.4 한글 사용자 경로 자동 우회

#### 문제 발생

kiwipiepy가 모델 파일을 로딩할 때 한글 경로를 못 다루는 버그가 있다.

```

(이어서 segfault)
```

#### 해결책

2. `kiwipiepy_model` 패키지의 모델 파일을 거기로 복사
3. `Kiwi(model_path=".kiwi_model")` 로 명시적 경로 지정

이 폴더는 [.gitignore](../.gitignore)에 추가되어 GitHub에는 안 올라간다.

처음 한 번만 복사하면 다음부터는 그 파일을 재사용하므로 시작 시간 영향 미미.

### 3.5 검증

실제 Supabase 130개 문서로 테스트.

**Query: "민원 신청 어떻게"**
- 벡터: 0.72 / 0.67 / 0.67
- BM25: 0 (일반어라 키워드 매칭 약함)
- 하이브리드: 0.700 / 0.655 / 0.651 → **벡터 결과 그대로 안전 폴백**

**Query: "노인 복지 지원"**
- 벡터: 0.76 / 0.55 / 0.55
- BM25: 11.36 / 11.09 / 6.20 → 강한 키워드 매칭
- 하이브리드: **1.000 / 0.802 / 0.653** → BM25가 1위를 명확하게 분리

키워드가 강할 때 BM25가 효과적으로 작동하고, 일반어일 때는 벡터 검색이 우세하게 작동하는 이상적인 결과.

## 4. 3단계: Flask → FastAPI 마이그레이션

### 4.1 동기

Flask의 한계:
1. **동기 처리 (WSGI)**: 한 요청 처리 중 다른 요청은 대기. LLM 호출(3~5초)이 길어 동시 사용자가 많아지면 응답 지연 누적.
2. **자동 API 문서 없음**: 어떤 엔드포인트가 있고 어떤 입출력인지 별도로 문서화해야 함.
3. **타입 검증 수동**: 요청 본문의 필드 누락/형식 오류를 수동 체크.

FastAPI 채택 이유:
- ASGI 기반 비동기 처리 (Starlette 위에 구축)
- Pydantic으로 자동 입출력 검증
- `/docs`에서 Swagger UI 자동 생성
- Flask와 비슷한 코드 스타일이라 마이그레이션 부담 적음

### 4.2 핵심 변경

[app.py](../app.py)를 전면 재작성. 동일한 엔드포인트(`/`, `/widget`, `/api/chat`, `/api/clear`, `/api/stats`)를 유지하되 내부 구조만 교체.

#### 4.2.1 Pydantic 요청/응답 스키마

```python
class ChatRequest(BaseModel):
    message: str = Field(..., min_length=1, max_length=500)

class ChatResponse(BaseModel):
    answer: str
    sources: list[Source]
    is_clarification: bool
```

이전 Flask 코드:
```python
data = request.get_json()
if not data or "message" not in data:
    return jsonify({"error": "메시지를 입력해주세요"}), 400
user_message = data["message"].strip()
if not user_message:
    return jsonify({"error": "빈 메시지입니다"}), 400
if len(user_message) > 500:
    return jsonify({"error": "메시지가 너무 깁니다 (최대 500자)"}), 400
```

이제 위 5줄은 Pydantic이 자동으로 처리한다. `message`가 누락이거나 500자 초과면 FastAPI가 알아서 422 Validation Error 반환.

#### 4.2.2 의존성 주입 — 관리자 인증

```python
async def require_admin(x_admin_key: Optional[str] = Header(default=None)):
    if not ADMIN_API_KEY:
        raise HTTPException(status_code=503, detail="...")
    if x_admin_key != ADMIN_API_KEY:
        raise HTTPException(status_code=401, detail="인증이 필요합니다")
    return True

@app.get("/api/stats")
async def stats(_auth: bool = Depends(require_admin)):
    ...
```

`Depends(require_admin)`로 인증 로직을 의존성으로 주입. 데코레이터 방식보다 테스트하기 쉽고 재사용 용이.

#### 4.2.3 세션 처리 — SessionMiddleware

Flask는 내장 `session` 객체가 있지만 FastAPI는 없다. Starlette의 `SessionMiddleware`로 대체:

```python
app.add_middleware(SessionMiddleware, secret_key=SECRET_KEY, same_site="lax")
```

`itsdangerous`로 서명된 쿠키 기반 세션. 동작은 Flask와 동일하지만 라이브러리만 다름.

#### 4.2.4 CORS / 보안 헤더

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=CORS_ALLOWED_ORIGINS,
    allow_credentials=True,
    ...
)

@app.middleware("http")
async def security_headers(request, call_next):
    response = await call_next(request)
    response.headers["X-Frame-Options"] = "SAMEORIGIN"
    response.headers["Content-Security-Policy"] = "..."
    ...
    return response
```

5/14 패치의 모든 보안 헤더 + CORS 정책을 그대로 유지.

#### 4.2.5 Rate Limiting — slowapi

`flask-limiter` 대신 `slowapi` 사용 (FastAPI용 포트):

```python
limiter = Limiter(key_func=get_remote_address, default_limits=["200 per hour"])
app.state.limiter = limiter
app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)

@app.post("/api/chat")
@limiter.limit(RATE_LIMIT_CHAT)
async def chat(request: Request, payload: ChatRequest):
    ...
```

#### 4.2.6 lifespan — APScheduler 통합

FastAPI는 앱 시작/종료 시점에 실행할 코드를 `lifespan` 컨텍스트로 정의:

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    # 시작 시
    await run_in_threadpool(get_chatbot)  # 모델 로딩
    _init_scheduler()                      # APScheduler 시작
    yield
    # 종료 시
    _scheduler.shutdown(wait=False)
```

이전에는 `main.py:run_web()`에서 BackgroundScheduler를 직접 띄우고 있었는데, FastAPI lifespan으로 일원화하면서 main.py의 스케줄러 코드를 제거. **중복 실행 위험 제거**.

#### 4.2.7 동기 호출의 비동기 위임 — run_in_threadpool

LangChain, Supabase 등 일부 라이브러리는 비동기 미지원. 동기 함수를 async 핸들러에서 그대로 호출하면 이벤트 루프가 멈춘다.

해결: `starlette.concurrency.run_in_threadpool`로 워커 스레드에 위임:

```python
@app.post("/api/chat")
async def chat(...):
    bot = get_chatbot()
    result = await run_in_threadpool(bot.chat, session_id, user_message)
    return ChatResponse(**result)
```

`bot.chat`은 동기 함수지만, 이벤트 루프를 막지 않고 다른 요청을 병렬 처리 가능.

#### 4.3.2 Starlette TemplateResponse 시그니처 변경

증상:
```
TypeError: unhashable type: 'dict'
File "app.py", line 217, in index
    return templates.TemplateResponse("index.html", {"request": request})
```

원인: Starlette 0.36+ 부터 `TemplateResponse`의 시그니처가 변경됨. `request`가 첫 번째 인자여야 함.

이전:
```python
templates.TemplateResponse("index.html", {"request": request})  # deprecated
```

수정:
```python
templates.TemplateResponse(request, "index.html")
```

## 9. 잔여 과제 (다음 단계 후보)

본 패치에서 의도적으로 보류한 작업.

### 9.1 ONNX 양자화 (NER 모델 가속)

NER 추론을 약 2~3배 가속(100~300ms → 30~100ms) 가능. 단:
- 정확도 1~3% 손실 가능성
- 디버깅이 까다로움
- 현재 응답 시간이 이미 실용 범위라 가성비 낮음

→ 발표 후 NER이 실제로 병목임이 확인되면 재검토.

### 9.2 silent failure를 사용자 안내 메시지로 노출

현재 검색/태깅 실패 시 로그만 남기고 빈 결과 반환. "일부 검색 결과를 가져오지 못했습니다" 같은 안내 메시지를 응답에 포함시켜 사용자 경험 개선.

### 9.3 BM25 인덱스 영속화

현재는 서버 기동 시 매번 메모리 인덱스 재구축. 문서가 수만 개로 늘어나면 시작 시간이 오래 걸린다. Supabase의 `tsvector` 컬럼으로 옮기거나 디스크 인덱스로 저장하는 방안 검토.

### 9.4 하이브리드 검색 가중치 튜닝

현재 0.7:0.3은 직관적 기본값. 실제 사용 데이터로 테스트셋을 만들어 정량 평가하여 최적 가중치 도출 필요.

### 9.5 멀티 워커 운영

현재는 단일 uvicorn 프로세스. 트래픽이 늘어나면 `uvicorn --workers 4`로 확장 가능하지만, 각 워커가 독립적인 챗봇/BM25 인덱스 인스턴스를 갖게 되므로 메모리 사용량이 4배가 된다. Redis 같은 공유 캐시 도입 검토.

## 10. 학생 발표용 핵심 메시지

본 패치 작업으로 강조할 만한 기술적 차별점:

1. **이중 PII 차단 (Defense in Depth)**: 정형 PII는 정규식으로 빠르게, 비정형 PII는 NER로 정밀하게. 두 방어선이 독립적이라 한쪽이 실패해도 다른 쪽이 작동.

2. **하이브리드 검색**: 단일 알고리즘이 아닌 의미 검색 + 키워드 검색의 가중 합산. 각 방식의 약점을 보완.

3. **프로덕션급 보안**: RLS, CORS, CSP, Rate Limit, 관리자 인증, 입출력 PII 마스킹까지 5/14 패치와 더해 7중 방어.

4. **현실 환경 대응**: 한글 사용자 경로 같은 실제 한국 개발 환경 문제를 코드에서 자동 우회.

5. **점진적 마이그레이션**: Flask에서 FastAPI로 옮길 때 동일한 엔드포인트/세션/보안 정책을 유지하면서 내부만 교체. 외부 사용자 영향 없음.
