## 3. 1차 수정: 증분 크롤링 후 BM25 인덱스 재구축

### 3.1 문제

검색은 3중 구조(메타필터 → 벡터 → BM25). 벡터는 Supabase 서버사이드라 즉시 최신을
반영하지만, **BM25는 웹 프로세스 메모리의 싱글턴**([bm25_index.py](../chatbot/bm25_index.py))이라
서버 기동 시 1회만 빌드된다.

스케줄러가 03:00에 같은 프로세스에서 `run_incremental`을 돌려도, 기존 코드는 벡터스토어만
갱신하고 BM25 싱글턴은 그대로였다.

→ 새로 추가/수정/삭제된 문서가 **서버 재시작 전까지 BM25 검색에 반영되지 않는 silent 버그.**

### 3.2 해결

[main.py](../main.py)에 `_rebuild_bm25_index()` 헬퍼 추가, `run_incremental` 마지막에
신규/수정/삭제가 하나라도 있으면 호출. `rebuild()`는 이미 구현돼 있었으므로 호출만 추가.

부수: 기존 "변경 없으면 조기 return" 구조 때문에 삭제만 발생한 경로가 BM25 갱신을
놓치던 문제도 함께 정리(if/else 구조로).

## 4. 1차 수정: LLM throttle 스레드 안전화

### 4.1 문제

[conversation.py](../chatbot/conversation.py)는 Groq 무료 티어 제한 준수용 "호출 간 4초"
throttle을 두는데, FastAPI가 `bot.chat`을 스레드풀로 위임하고 봇이 싱글턴이므로
**여러 워커 스레드가 락 없이 `_last_call_time`을 동시 read/write**.

- 두 스레드가 동시에 `elapsed < 4`를 통과 → 4초 간격 깨짐, Groq 제한 초과 가능
- 갱신 타이밍 경합으로 간격 계산 기준 불안정

### 4.2 해결

`threading.Lock`으로 **간격 계산·갱신만** 보호. 락 안에서 `_last_call_time = time.time()`까지
끝낸 뒤 락을 풀고, 실제 네트워크 호출(`invoke`)은 락 밖에서 수행.

- 연속 호출이 4초 이상 간격으로 **시작**되도록 보장(락 구간 직렬화)
- 느린 LLM 응답을 기다리는 동안 락을 안 들고 있어, throttle 대기만 직렬 + 네트워크는 병행

## 5. 2차 수정: HTTP 조건부 GET 기반 증분 크롤링

### 5.1 문제

PDF의 ① 요구사항은 "변경된 페이지만 주기적으로 감지·업데이트하여 **서버 부하 최소화**".
하지만 기존 `run_incremental`은:

1. 모든 메뉴를 처음부터 BFS로 재크롤링 (300+ 페이지 매번 전수 다운로드)
2. 다운로드된 본문의 content_hash를 DB와 비교
3. 변경분만 재처리

→ **DB/임베딩 비용은 절감되지만, 대상 서버(saha.go.kr)의 트래픽 부하는 전혀 줄지 않음.**
"서버 부하 최소화" 취지를 절반만 달성.

### 5.2 해결 — ETag/Last-Modified 조건부 GET

#### 구조 변경

```
[기존]                          [신규]
크롤러 → GET 모든 URL           DB에서 url별 ETag/Last-Modified 로드
        ↓                       크롤러에 주입 (set_cache_validators)
       전체 본문 다운로드               ↓
        ↓                       각 URL에 If-None-Match / If-Modified-Since
       hash 비교                       ↓
                                ┌─────┴─────┐
                                ↓           ↓
                              304          200
                          (본문 없음)   (본문 있음)
                          → unchanged   → hash 비교 후 new/updated/unchanged
```

#### 핵심 코드 흐름

[crawler/saha_crawler.py](../crawler/saha_crawler.py):

```python
class SahaCrawler:
    def __init__(self, ...):
        self.cache_validators: dict = {}  # {url: {"etag", "last_modified"}}

    def set_cache_validators(self, validators: dict):
        self.cache_validators = validators or {}

    def fetch_page(self, url: str) -> Optional[dict]:
        cached = self.cache_validators.get(url, {})
        cond_headers = {}
        if cached.get("etag"):
            cond_headers["If-None-Match"] = cached["etag"]
        if cached.get("last_modified"):
            cond_headers["If-Modified-Since"] = cached["last_modified"]

        resp = self.session.get(url, headers=cond_headers, ...)
        if resp.status_code == 304:
            return {"html": None, "status": "not_modified", ...}
        if resp.status_code == 404:
            return {"html": None, "status": "deleted", ...}
        # 200: return html + new ETag/Last-Modified
        return {"html": resp.text, "etag": resp.headers.get("ETag"),
                "last_modified": resp.headers.get("Last-Modified"), "status": "ok"}
```

[main.py](../main.py) `run_incremental`:

```python
validators = db.get_cache_validators()   # {url: {etag, last_modified, category}}
crawler.set_cache_validators(...)
known_by_menu = {menu: [urls in that category]}

for menu_name, menu_path in menus.items():
    pages = crawler.crawl_menu(menu_name, start_url,
                                max_pages=MAX_PAGES_PER_MENU,
                                known_urls=known_by_menu.get(menu_name, []))
    for page in pages:
        if page.transient_fail: continue          # 보존
        if page.deleted:        db.delete_page(...); stats["deleted"] += 1
        if page.not_modified:   stats["unchanged"] += 1   # 본문 다운로드 없음
        else:                   db.upsert_raw_page(page)  # hash로 한 번 더 확인
```

#### 설계상 결정 사항

- **이중 확인(304 + content_hash)**: 서버가 ETag/Last-Modified를 안 주거나 부정확할 수
  있어, 200을 받아도 content_hash로 한 번 더 비교한다. 안전망.
- **known URL 시드**: BFS 큐에 `start_url` 외에 해당 카테고리의 known URL을 함께
  넣어, landing 페이지가 304여도 모든 알려진 URL이 점검된다.
- **`max_pages` 의미 재정의**: BFS로 **새로 발견한** URL 수의 상한으로만 적용.
  known URL은 별도 집계되어 항상 시도(증분 누락 방지).
- **304 페이지의 링크는 미추출**: 본문이 없으니 불가피. 신규 페이지는 listing 페이지가
  갱신될 때 자연 발견되므로 누락 위험은 낮음. (saha.go.kr CMS는 신규 항목 추가 시
  listing 본문이 함께 변하는 구조)
- **orphan 정리 정책**: `404` 응답 + `known URL 중 이번 회차에 한 번도 방문되지
  않은 것`을 삭제. 일시 네트워크 오류로 인한 오삭제 방지.

#### Database 변경

[database_db/database.py](../database_db/database.py):

- `get_cache_validators()`: 전체 raw_pages에서 (etag, last_modified, category)를 dict로 반환
- `save_raw_page` / `upsert_raw_page`: ETag/Last-Modified도 함께 저장
- `_update_cache_validators_if_present`: 본문 미변경이어도 새 검증자가 있으면 갱신

### 5.3 효과 — 실측 결과: **대상 서버에서 무효**

2026-05-31 실측 결과, **saha.go.kr이 ETag/Last-Modified 헤더를 일절 제공하지 않음**.
응답 헤더 직접 확인:

```
HTTP/1.1 200 OK
Cache-Control: no-cache, no-store, must-revalidate
Expires: 0
ETag: (없음)
Last-Modified: (없음)
```

`Cache-Control: no-store, must-revalidate`는 서버가 모든 캐싱을 명시적으로 거부한다는
뜻이며, 이런 응답에서는 `If-None-Match` / `If-Modified-Since`를 보내도 영원히 304가
오지 않는다(검증할 값이 없음).

1차 `python main.py --mode incremental` 실행 결과:

- 425페이지 모두 200 OK
- raw_pages.etag, raw_pages.last_modified 컬럼 181행 모두 NULL
- 304 카운트 0

#### 의미

- 이 패치의 조건부 GET 구조 자체는 정상 작동(코드·DB 마이그레이션·헤더 전송 모두 OK)
- 그러나 **대상 서버가 캐시 검증자를 안 주므로 다운로드 트래픽 절감 효과는 실질 0%**
- 일지·발표자료에 "95% 절감" 같은 수치는 쓰면 안 됨

#### 그래도 남는 가치

- **DB·임베딩·LLM 태깅 비용 절감**: content_hash 비교로 본문 미변경분은 재처리·재임베딩
  하지 않는 부분은 그대로 유효(이건 5/28 패치 이전부터 작동했던 부분)
- **삭제 동기화**: 404 + orphan 정리로 사라진 페이지를 DB에서 제거하는 로직은 신규 가치
- **인프라 호환성**: ETag/Last-Modified를 주는 다른 출처(예: 보도자료 RSS, 외부 데이터셋)를
  추가했을 때 그대로 활용 가능

### 5.4 대안 — 다음 백로그

`Cache-Control: no-store`인 서버에서 진짜 트래픽 절감을 하려면:

1. **listing/landing 페이지만 우선 폴링** → 변경 감지된 listing에서 링크된 상세만 본문 GET
   (saha.go.kr CMS는 신규 항목 추가 시 listing이 함께 갱신되는 구조 추정 — 검증 필요)
2. **HEAD 요청 활용**: 본문 다운로드 없이 Content-Length만 받아 본문 변화 감지(이 서버가
   HEAD에서 의미 있는 응답을 주는지는 별도 검증 필요)
3. **변경 주기 학습**: URL별 평균 변경 주기를 통계로 잡아 자주 변하는 URL만 잦은 주기로 폴링

## 6. 2차 수정: 담당부서(department) 자동 태깅

### 6.1 문제

PDF의 ① 요구사항: "AI 자동 태깅: LLM을 활용해 수집 데이터의 메타데이터
(**카테고리, 부서, 핵심 키워드** 등)를 자동 추출 및 태깅."

기존 태깅: `service_type`, `target_audience`, `keywords`, `has_deadline`,
`has_contact_info`, `summary` → **부서 누락**.

### 6.2 해결

전 계층에 `department` 필드를 흘려보냄:

| 계층 | 변경 |
|------|------|
| LLM 프롬프트 | `BATCH_TAGGING_PROMPT` / `SINGLE_TAGGING_PROMPT`에 `"department"` 추출 지시. 본문 미명시 시 `null`. |
| DB 스키마 | `processed_chunks.department text` 컬럼 + 인덱스 |
| Database.save_chunks_bulk | `department` 컬럼에 저장 |
| 벡터 메타(JSONB) | `documents.metadata.department` 자동 포함 (스키마 변경 불요) |
| Retriever.format_context | 출처 dict에 `department` 포함 |
| FastAPI Source 스키마 | `department: str = ""` 필드 추가 |
| 프론트 출처 카드 | 부서 배지 표시(`.source-badge-dept`, slate 색상) |

LLM이 부서를 보수적으로 추출하도록 프롬프트에 "본문에 명시되지 않았으면 null" 명시.
환각된 부서명이 UI에 노출되는 것을 방지.

### 6.3 효과

- 사하구청 홈페이지는 대부분의 상세 페이지에 "담당부서: XX과" 표기 존재 → 추출률 양호 예상
- 출처 카드에 "복지정책과" 등 부서가 함께 표시되어, 구민이 추가 문의 시 정확한 부서로 연락 가능
- 향후 부서별 검색 필터 / 부서별 FAQ 통계 등 확장 기반

## 7. 마이그레이션 적용 방법

Supabase Dashboard > SQL Editor에서 다음 컬럼이 추가됐는지 확인. 없으면 실행:

```sql
-- 조건부 GET용 캐시 검증자
alter table raw_pages add column if not exists etag text;
alter table raw_pages add column if not exists last_modified text;

-- 담당부서 태깅
alter table processed_chunks add column if not exists department text;
create index if not exists ix_chunks_department on processed_chunks(department);
```

[setup_supabase.sql](../setup_supabase.sql) 본문에도 동일 ALTER 문을 포함해두어
신규 환경에서도 자동 적용된다.

### 데이터 백필

- **etag/last_modified**: 첫 회차 `run_incremental` 실행 시 200 응답으로부터 채워짐
  (그 회차는 전수 다운로드, 다음 회차부터 304 절감 효과 발생)
- **department**: 기존 청크는 NULL 상태. 전체 재태깅 또는 변경된 페이지부터 자연 채워짐.
  필요 시 `python main.py --mode process`로 전수 재태깅 가능 (Groq 무료 티어 RPM 고려).
