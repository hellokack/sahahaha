# 운영 안정화 패치 작업 기록 (2026-05-22)

> 5/17 고도화 패치([patch_2026-05-17.md](patch_2026-05-17.md)) 이후 발표 준비 단계에서
> 식별된 운영 관점 약점(silent failure, 가중치 근거 부재, 발표용 정량 자료 부재)을
> 한 번에 보완한 작업.

---

## 1. 개요

| 단계 | 주제 | 효과 |
|:---:|------|------|
| 1 | degraded 신호 + 프론트 배너 | 검색·LLM 부분 실패 시 사용자에게 신뢰도 저하 명시 |
| 2 | 평가 테스트셋(50개) + 가중치 그리드 서치 스크립트 | `0.7:0.3`을 "근거 있는 기본값"으로 만드는 도구 |
| 3 | 발표용 벤치마크 표 문서 | 응답시간·PII 차단율·보안 체크리스트 한 슬라이드 분량 |

설계 원칙:
- 이미 구현된 기능을 "보여주는/지키는" 작업에 집중. 새 알고리즘 도입은 없음.
- 의도적으로 보류한 것: Redis 공유 캐시, 쿼리 유형별 동적 가중치 분류기
  (현재 트래픽 규모에서 가성비 낮음 — 추후 사용량이 늘면 재검토).

## 2. 1단계: degraded 신호 파이프라인

### 2.1 문제 정의

5/14 / 5/17 작업으로 retriever와 LLM 호출 곳곳에 `try/except + 로그 + 빈 결과 반환`
패턴이 깔려 있었다. 이 구조는 서버 입장에선 안전하지만 사용자 입장에서는:

- 벡터 RPC 실패 → 빈 검색 결과 → "해당 정보를 찾지 못했습니다" 평소 답변과 구분 안 됨
- BM25 인덱스 비활성 → 벡터 단독 검색 결과를 받음 (정확도 낮음). 사용자는 모름
- LLM 호출 실패 → 정형 문구의 사과 답변. 시스템 결함인지 알 수 없음

→ 사용자가 부정확한 답변을 정상 답변으로 오인할 위험.

### 2.2 해결 구조

```
검색/LLM 단계
   ├── 정상: {results, degraded: False, reason: None}
   └── 부분 실패: {results, degraded: True, reason: "..."}
                         ↓
            ChatBot.chat()이 신호 전파
                         ↓
              ChatResponse(degraded, degraded_reason)
                         ↓
          프론트가 노란 배너로 시각적 안내
```

### 2.3 변경 파일

#### [chatbot/retriever.py](../chatbot/retriever.py)

`HybridRetriever.search()` 반환을 `list[dict]` → `dict`로 변경.

```python
def search(self, query: str, k: int = 5) -> dict:
    """
    Returns:
        {
            "results": list[dict],
            "degraded": bool,
            "reason": str | None,
        }
    """
```

degraded=True 케이스:

| reason | 의미 |
|--------|------|
| `vector_search_failed` | 벡터 RPC 호출이 예외로 실패 (결과 없음) |
| `bm25_failed` | BM25 인덱스가 비활성. 벡터 결과만으로 응답 → 정확도 평소보다 낮음 |

"결과 0건"은 degraded로 보지 않는다 — 시스템 결함이 아니라 매칭 문서가 없는 정상 동작.

#### [chatbot/conversation.py](../chatbot/conversation.py)

`ChatBot.chat()`이 degraded 신호를 받아 응답 dict에 포함.
LLM 호출이 실패하면 `degraded=True, reason="llm_failed"`를 추가로 설정.

PII 차단 early return 경로에는 `degraded=False`로 명시 (정상 동작).

#### [app.py](../app.py)

```python
class ChatResponse(BaseModel):
    answer: str
    sources: list[Source]
    is_clarification: bool
    degraded: bool = False
    degraded_reason: Optional[str] = None
```

`/api/chat` 500 응답에도 `degraded=True, degraded_reason="internal_error"` 추가.

#### [static/js/chat.js](../static/js/chat.js)

원인 코드 → 사용자 문구 매핑:

| reason | 안내 문구 |
|--------|----------|
| `vector_search_failed` | "검색 시스템 일부에 문제가 발생해 평소보다 결과가 적거나 부정확할 수 있습니다." |
| `bm25_failed` | "키워드 보정 검색이 비활성화된 상태입니다. 답변 정확도가 평소보다 낮을 수 있습니다." |
| `llm_failed` | "답변 생성 중 일시적인 오류가 발생했습니다. 잠시 후 다시 시도해주세요." |
| `internal_error` | "일시적인 시스템 오류가 발생했습니다. 잠시 후 다시 시도해주세요." |

봇 메시지 렌더링 시 답변 버블과 출처 카드 사이에 `degraded-banner` 삽입.

#### [static/css/style.css](../static/css/style.css)

노란 톤(`#FFF8E1` 배경 + `#FF9800` 좌측 강조선) 경고 배너 스타일. 다크모드(`prefers-color-scheme: dark`)
에서는 어두운 황갈색 배경 + 밝은 텍스트로 명도 대비 확보.

### 2.4 검증 방법

```bash
# 시나리오 1: BM25 비활성 (Kiwi 로딩 실패 시뮬레이션) → bm25_failed 배너
# 시나리오 2: Supabase 네트워크 차단 후 호출 → vector_search_failed 배너
# 시나리오 3: GROQ_API_KEY 잘못된 값으로 변경 후 호출 → llm_failed 배너
```

각 케이스에서 답변 카드 상단에 노란 배너가 표시되어야 한다.

## 3. 2단계: 평가 테스트셋 + 그리드 서치

### 3.1 동기

5/17 시점에 도입한 가중치 `HYBRID_VECTOR_WEIGHT=0.7`, `HYBRID_BM25_WEIGHT=0.3`은
직관적 기본값이었다. 발표에서 "왜 0.7:0.3인가요?"를 물었을 때 근거가 없었다.
실제 측정 인프라를 만들어 "근거 있는 0.7:0.3"으로 바꾼다.

### 3.2 평가셋 — [evaluation/testset.json](../evaluation/testset.json)

50개 쿼리, 사하구청 도메인(민원/복지/세금/환경/문화/일반안내/모호어). 각 쿼리에:

```json
{
  "id": "Q11",
  "query": "노인 복지 지원 어떤거 있어요",
  "expected_keywords": ["노인", "어르신", "복지"],
  "type": "keyword"
}
```

`type`은 `keyword`(단어 매칭 강함) / `general`(일반어, 의미 검색 우세) 구분용.
가중치 조합이 어느 쿼리 유형에 유리한지 분석할 수 있게 라벨링.

#### 판정 규칙

검색된 문서의 `(title + content)` 안에 `expected_keywords` 중 하나 이상 포함되면 정답.

chunk_id 단위 라벨링은 의도적으로 피했다 — Supabase 청크가 갱신될 때마다
라벨 유지보수 부담이 크다. 키워드 판정은 다소 느슨하지만 가중치 **상대 비교**에는 충분.

### 3.3 그리드 서치 스크립트 — [evaluation/grid_search.py](../evaluation/grid_search.py)

흐름:

1. testset 로딩
2. **쿼리당 후보 풀을 1회만 수집** — 벡터 상위 20건 + BM25 상위 30건
3. 가중치 8조합(0.3:0.7 ~ 1.0:0.0)에 대해 캐시된 후보를 재랭킹·평가

후보 풀 재사용 덕분에 가중치 조합이 늘어나도 벡터 RPC·BM25 토큰화는 50회로 고정.
실행 비용은 사실상 가중 합산 + 정렬뿐.

#### 측정 지표

- `Recall@k` (k=1, 3, 5): 상위 k개 안에 정답이 1건 이상 포함된 비율
- `MRR` (Mean Reciprocal Rank): 정답이 처음 나오는 순위의 역수 평균

#### 출력

`evaluation/results.md`에 markdown 표로 저장 + 표준 출력에 MRR/Recall@5 최적 가중치 출력.

### 3.4 사용법

```bash
python -m evaluation.grid_search
```

`.env`가 정상 설정된 상태(Supabase + Groq + Kiwi 모델 준비)에서 실행. 50쿼리 × 8조합 그리드.

### 3.5 한계

- 50건은 통계적으로 견고하지 않다. 운영 후 실제 사용자 질의로 150건+ 확장 권장.
- 키워드 판정은 chunk_id 직접 라벨링 대비 1~2% 과대평가 가능. 가중치의
  **상대 순위**가 일치하면 무방.

## 6. 운영 영향

- 기존 정상 경로의 응답 시간/메모리에는 영향 없음 (`degraded=False`가 기본).
- 5/17 시점의 `chat()` 반환 키 3개를 사용하는 외부 호출자가 있다면 키 2개가 추가된다 —
  app.py의 `ChatResponse(**result)`가 유일한 호출자라 영향 없음.
- 그리드 서치는 평가용 별도 스크립트라 서버 코드 경로와 분리. 운영 부담 없음.

## 7. 잔여 과제

본 패치에서도 의도적으로 보류한 작업.

### 7.1 Redis 공유 캐시 도입

`uvicorn --workers 4` 환경에서 NER·BM25·임베딩 모델이 워커마다 중복 적재되어
메모리 4배 증가 문제가 있다. 단, 현재 동시 접속이 적은 프로토타입 단계에서는
단일 워커로도 충분 — 사용량이 늘어나는 시점에 재검토.

### 7.2 쿼리 유형별 동적 가중치

50건 평가에서 `0.6:0.4`와 `0.7:0.3`이 사실상 동률이라면 분류기 도입 효과가 작을
가능성이 크다. 평가셋을 150건+로 확장한 뒤 효과 검증 후 결정.

### 7.3 ONNX 양자화 (NER 가속)

현재 응답 시간이 실용 범위라 우선순위 후순위. NER이 명확한 병목임이 측정으로
확인되면 진행.

### 7.4 BM25 인덱스 영속화

130문서 규모에서는 빌드 시간이 약 5~15초로 부담 없음. 수만 문서 단계에서 재검토.
